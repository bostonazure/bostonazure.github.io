Managing Speaker Queue with Sessionize

Sessionize has 5 states:

  1. Nominated - new submissions arrive in this state - for Boston Azure these are typically self-nominations
  2. Accept Queue - we'd like to do this talk, but have not confirmed a date with the speaker
  3. Accepted - scheduled
  4. Decline Queue - usually this is coupled with one Accepted Queue item and communications can be bundled
  5. Declined - and after notified this one is declined

So here's a normal way for this to play out: speaker submits three talks; one is selected and goes into the Accept Queue, while two go into the Decline Queue; a communication to this effect ("hey, we'd like to proceed with this talk, and here are a few proposed dates"); once a date is confirmed, the main talk can move to the Accepted state and the others can go into Declined state.

Organizer can see which speakers need attention from the Sessionize dashboard for Boston Azure.
